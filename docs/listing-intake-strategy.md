# Стратегия добавления объявлений

Проект поддерживает несколько способов попадания объявлений в анализ. Сейчас это концептуальный слой и sample/mock данные: реальный парсинг URL, scraping и подключение внешних источников не реализованы.

## 1. Автоматический мониторинг по поисковому профилю

Поисковый профиль описывает набор фильтров: бюджет, тип помещения, этаж, год дома, район, признаки арендатора и другие параметры. Такой режим нужен, когда инвестор хочет регулярно получать новые подходящие объекты по заранее заданной стратегии.

Источник таких объектов должен быть помечен как автоматический мониторинг, а сами объявления должны хранить связь с поисковым профилем.

## 2. Мониторинг по локации

Локационный мониторинг нужен, когда инвестор следит за конкретной зоной: районом, улицей, жилым комплексом, станцией метро или радиусом вокруг точки. Этот режим полезен для сравнения объектов внутри одной торговой зоны и для наблюдения за локальной конкуренцией.

## 3. Мониторинг по источнику или платформе

Платформенный мониторинг нужен, когда важно отслеживать конкретный источник: CIAN, Avito, Domclick, Yandex Realty, партнерский feed или внутреннюю базу. Такой режим помогает контролировать качество источников и понимать, где объявление было найдено.

## 4. Ручной URL / ручная подборка

Ручное добавление нужно, когда пользователь сам нашел одно или несколько объявлений и хочет отправить их в анализ без настройки постоянного мониторинга.

Ручной intake должен поддерживать:

- добавление одного URL;
- добавление нескольких URL;
- создание ручной подборки;
- отдельное отображение ручных подборок от автоматически собранных объявлений;
- явную маркировку источника как `manual`.

Ручной URL intake полезен, когда:

- инвестор нашел объявление самостоятельно;
- инвестор хочет сравнить только выбранные объявления;
- пользователь хочет быстро проверить объект перед добавлением локации или профиля в мониторинг;
- источник еще не поддерживается автоматическим парсингом.

## TASK-012 API ручного intake

Backend поддерживает сохраняемый intake без scraping:

- `POST /api/v1/manual-intake/batches` создаёт подборку из 1–100 уникальных URL;
- `GET /api/v1/manual-intake/batches` возвращает сохранённые подборки;
- `GET /api/v1/manual-intake/batches/{batch_id}` возвращает одну подборку;
- URL CIAN, Avito, Domclick и Yandex Realty получают статус `queued`;
- URL неподдерживаемого источника сохраняется со статусом `failed` и объяснением;
- batch получает статус `failed`, если в нём нет ни одного поддерживаемого URL.

Источник определяется только по границам hostname. Например,
`cian.ru.attacker.example` не считается CIAN. Создание batch не выполняет HTTP-запрос
к площадке, не обходит login, captcha или anti-bot protection и не извлекает данные
из страницы. Оно создаёт контролируемую очередь для будущих разрешённых adapters.

## Data model

```text
manual_intake_batch:
  id
  name
  description
  created_at
  status
  urls
  total_urls
  processed_count
  failed_count
  analyzed_count
  source: "manual"
  linked_search_profile_id optional

manual_listing_url:
  id
  url
  source_detected
  status
  error_message optional
  created_at
```

Статусы:

- `draft`;
- `queued`;
- `processing`;
- `analyzed`;
- `failed`.

Важно: backend уже сохраняет реальные batch и URL через API, но dashboard пока
показывает sample-список. Парсинг страниц и обращения к внешним площадкам не
реализованы.

## Гибкие поисковые профили

TASK-008B добавляет sample-концепт нескольких поисковых профилей. Профиль задает стартовые значения фильтров и связывает intake funnel с конкретной инвестиционной стратегией.

Предопределенные профили:

- `Коммерция 100–400 млн`;
- `Малые помещения до 30 млн`;
- `Офисы 30–150 млн`;
- `Помещения с арендаторами`;
- `Пользовательский профиль`.

Профиль хранит `default_filters`: диапазон цены, цену за м², площадь, этаж, первый этаж, локацию, источник, категорию, тип сделки, рекомендацию, минимальный инвестиционный рейтинг, максимальный риск, минимальное качество данных, признаки арендатора, доходность, мощность, год здания, наличие фото, недостающую информацию и уровень поддержки рынка.

## Intake funnel по профилю

Для каждого профиля sample API возвращает воронку:

- активный профиль;
- последний запуск;
- следующий запуск;
- интервал обновления;
- максимальное число объявлений на источник;
- разбивку по источникам;
- разбивку по локациям.

Это не запускает scraping. Воронка показывает, как будущий мониторинг будет объяснять объем входящих объявлений по выбранному профилю.
